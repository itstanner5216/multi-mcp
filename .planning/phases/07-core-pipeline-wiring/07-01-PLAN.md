### 7. Conversation query extraction (F-03)

**Files:** `src/multimcp/mcp_proxy.py`, `src/multimcp/retrieval/pipeline.py`
**Source plan (line 296):** "Extract conversation terms (stopword removal, noun phrases, tool verbs)"

**Producer contract — mcp_proxy.py `_list_tools()`:**

The proxy must build `conversation_context` from exactly these three sources in this order:
1. Session tool history — tool names called earlier in this session, in call order
2. Argument keys — keys from recent tool call argument dicts, in call order
3. Router describe targets — tool names the model asked to describe via the routing tool, in call order

```python
session_id = self._get_session_id()
conv_parts = []
conv_parts.extend(self.retrieval_pipeline.get_session_tool_history(session_id))
conv_parts.extend(self.retrieval_pipeline.get_session_argument_keys(session_id))
conv_parts.extend(self.retrieval_pipeline.get_session_router_describes(session_id))
conversation_context = " ".join(conv_parts)
tools = await self.retrieval_pipeline.get_tools_for_list(session_id, conversation_context)
```

**Consumer contract — pipeline.py `_extract_conv_terms(raw: str) -> str`:**

This method applies the following deterministic extraction in this exact order:
1. Lowercase the entire input
2. Replace `_` and `-` with spaces
3. Regex-tokenize with pattern `[a-z0-9]+` (only alphanumeric tokens)
4. Remove stopwords: `{"the", "a", "an", "is", "are", "was", "were", "be", "to", "of", "and", "in", "for", "on", "with", "true", "false", "null", "none"}`
5. Deduplicate preserving first occurrence
6. Generate adjacent non-stopword bigrams (pairs of consecutive tokens joined by space) and append them
7. For each token, if it matches any entry in this fixed action-verb lexicon, append the aliases:
   - `list` → `get fetch show enumerate`
   - `create` → `add new make insert`
   - `search` → `find query lookup`
   - `delete` → `remove destroy drop`
   - `update` → `edit modify change patch`
   - `run` → `execute invoke start`
   - `get` → `fetch read retrieve`
8. Deduplicate the final token list preserving first occurrence
9. Return space-joined result

**Pipeline stores (populated by `on_tool_called()`):**
- `_session_tool_history: dict[str, list[str]]` — tool names in call order
- `_session_arg_keys: dict[str, list[str]]` — argument key names in call order
- `_session_router_describes: dict[str, list[str]]` — tool names the model requested to describe via routing tool, in call order

**Tier 6 selection rule (exact deterministic algorithm):**

Namespace priority order (defined in `static_categories.py` as `TIER6_NAMESPACE_PRIORITY`):
1. `filesystem`
2. `shell`
3. `web_search`
4. `github`
5. `docker`
6. `npm`
7. `pip`
8. `cargo`
9. `kubectl`
10. `terraform`
11. `slack`
12. `context7`

For each namespace in order, select the lexicographically smallest matching registry key not yet selected. If fewer than 12 tools are selected after the namespace pass, fill remaining slots with lexicographically smallest remaining registry keys. Expose at most 12 direct tools. All remaining tools go to the routing-tool enum. The routing tool does not count toward the 12.

**Tier 4 classification rule (exact precedence, first match wins):**

1. `infrastructure` if telemetry tokens contain any of: `infra:terraform`, `infra:kubernetes`, `manifest:Chart.yaml`
2. `rust_cli` if tokens contain: `manifest:Cargo.toml` or `lang:rust`
3. `python_web` if tokens contain: `manifest:pyproject.toml` or `lang:python`
4. `node_web` if tokens contain: `manifest:package.json` or `lang:javascript` or `lang:typescript`
5. `generic` if `workspace_confidence >= 0.45`
6. Tier 4 does not fire; fall through to Tier 5

`_static_category_defaults(project_type)` must:
1. Look up `STATIC_CATEGORIES[project_type]`
2. Collect tools from the registry whose namespace matches entries in the `always` list, then `likely` list
3. Cap the result at `dynamic_k`
4. Return as `list[ScoredTool]` with descending score (always tools score 1.0, likely tools score 0.8)

**Tier 5 frequency prior rule (exact implementation):**

Source: `FileRetrievalLogger` JSONL at configured `log_path`.
Time window: last 7 days only.
Exclude events whose `group` field is `"shadow"`.

Usage events counted:
- `direct_tool_calls` entries: each list element contributes one count. If the same tool appears multiple times within one event's `direct_tool_calls` list, each occurrence counts separately.
- `router_proxies` entries: each list element contributes one count. Same rule: multiple occurrences within one event's list each count separately.
- Cross-list: if a tool appears in both `direct_tool_calls` and `router_proxies` within the same event, each source contributes its count independently — a tool present in both lists counts twice for that event.

Do NOT use `active_tool_ids` as usage frequency.

If no log file exists, no events within 7 days, or no nonzero scores, Tier 5 does not fire — fall through to Tier 6.

`NullLogger` produces no log file, so Tier 5 always falls through when `NullLogger` is in use.
