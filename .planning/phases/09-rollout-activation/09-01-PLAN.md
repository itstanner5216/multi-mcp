# Plan 09-01: Rollout Activation & Observability

## Objective

Replace the temporary hardcoded shadow-mode retrieval bootstrap in `multi_mcp.py` (the `_make_startup_retrieval_config()` helper, which forces `enabled=True, shadow_mode=True, rollout_stage="shadow"` at every startup) with YAML-driven `RetrievalConfig` sourced from `yaml_config.retrieval` (`MultiMCPConfig`). Wire the shadow/canary rollout into the server runtime and fix cutover gates to compute correct metrics per source plan.

After this plan:
- No hardcoded rollout state remains in the app bootstrap; all retrieval behavior is config-driven via YAML.
- When no `retrieval:` block is present in YAML, `RetrievalSettings()` model defaults apply (`enabled=False`, `shadow_mode=False`), preserving full backward compatibility.
- The current temporary shadow-mode live wiring is removed.
- Operators can enable, shadow-test, and promote the pipeline through YAML config without code changes.

### 7. Assert pipeline runtime semantics for shadow/canary/GA (F-09)

**File:** `src/multimcp/retrieval/pipeline.py`

The audit's core F-09 finding is that the rollout infrastructure is "well-tested but disconnected." Config plumbing alone (changes 1-2) does not close F-09 if the pipeline's runtime behavior for each rollout stage is not verified to work correctly.

**Mandatory implementation contract — `pipeline.py` `get_tools_for_list()` dispatch:**

| Stage | `enabled` | `shadow_mode` | `rollout_stage` | Pipeline behavior |
|-------|-----------|---------------|-----------------|-------------------|
| Off | False | * | * | Return all tools (passthrough). Short-circuit at line 67-69. |
| Shadow | True | True | shadow | Score via full BMXF+RRF path. Log RankingEvent. Return all tools. |
| Canary | True | False | canary | Control sessions: return all tools. Canary sessions: return bounded set + routing tool. |
| GA | True | False | ga | All sessions: return bounded set + routing tool. |

**Required patches to `pipeline.py`:**
- Shadow mode: `is_filtered` must be forced to `False` when `shadow_mode=True`, evaluated before the `rollout_stage` check. The shadow mode guard must be the first dispatch branch.
- Canary mode: `get_session_group()` from `rollout.py` must be called with the real `session_id` provided by Phase 8. The control branch returns all tools. The canary branch returns the bounded set + routing tool.
- GA mode: All sessions take the `is_filtered=True` branch. The bounded set + routing tool path must execute for every session.

**Mandatory dispatch order in `get_tools_for_list()`:**

The existing dispatch is already correct in the current branch. Verify it matches this contract exactly:

```python
# Step 1: Kill switch — Off mode, no scoring
if not self.config.enabled:
    return [m.tool for m in self.tool_registry.values()]  # list[types.Tool]

# Determine session group ONCE (used by canary dispatch below)
group = get_session_group(session_id, self.config)  # passes full RetrievalConfig, NOT canary_percentage

# ... (scoring computation runs here) ...

# Shadow mode is the FIRST dispatch guard — precedes rollout_stage check
if self.config.shadow_mode:
    is_filtered = False                        # Shadow: score+log, return all tools
elif self.config.rollout_stage == "ga":
    is_filtered = True                         # GA: all sessions get bounded set
elif self.config.rollout_stage == "canary":
    is_filtered = (group == "canary")          # Canary: only canary-group sessions filtered
else:                                          # "control" or unknown: passthrough
    is_filtered = False

if is_filtered:
    # CANARY/GA: return bounded active set + optional routing tool (list[types.Tool])
    # actual assembly uses self.ranker/self.assembler path or direct registry lookup
    result = <bounded active set + routing_tool if configured>
else:
    # SHADOW/CONTROL: return all tools
    result = [m.tool for m in self.tool_registry.values()]

return result  # always list[types.Tool]
```

**Key contract points:**
- `get_session_group` takes `(session_id: str, config: RetrievalConfig)` — NOT `canary_percentage` alone.
- Off-mode return: `[m.tool for m in self.tool_registry.values()]` (extracts `types.Tool` from each `ToolMapping`, not the `ToolMapping` itself).
- Passthrough (shadow/control) return: same shape — `[m.tool for m in self.tool_registry.values()]`.
- Shadow guard must be the FIRST dispatch branch (precedes `rollout_stage` check).
- `group` is resolved once from the full config before the dispatch tree, not per-branch.
